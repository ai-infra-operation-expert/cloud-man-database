# 贡献指南

感谢您对CloudMaster项目的兴趣和支持！为了确保项目的质量和一致性，我们制定了以下贡献指南，请在提交贡献前仔细阅读。

## 开发流程

1. **Fork项目**：在GitHub上fork项目到您的个人账号
2. **克隆代码**：将fork后的仓库克隆到本地
3. **创建分支**：为您的贡献创建一个新的分支
4. **开发功能**：实现您的功能或修复bug
5. **运行测试**：确保所有测试通过
6. **提交代码**：使用规范的提交信息提交代码
7. **创建PR**：向主仓库提交Pull Request

## 代码规范

### TypeScript

- 使用TypeScript编写所有代码
- 严格遵循TypeScript类型检查
- 使用接口定义数据模型
- 使用类型断言时要谨慎

### 代码风格

- 使用ESLint进行代码检查
- 遵循项目的ESLint配置
- 使用Prettier进行代码格式化
- 保持代码简洁、清晰、易读

### 组件开发

- 组件命名使用PascalCase
- 函数命名使用camelCase
- 组件职责单一，功能明确
- 提供完整的TypeScript类型定义
- 添加必要的注释

## 提交规范

提交信息应遵循以下格式：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Type

- **feat**: 新功能
- **fix**: 修复bug
- **docs**: 文档更新
- **style**: 代码风格变更
- **refactor**: 代码重构
- **test**: 测试相关
- **chore**: 构建过程或辅助工具的变动

### Scope

- 组件名称
- 页面名称
- 功能模块名称

### Description

- 简短、清晰地描述您的变更
- 使用命令式语气（例如："添加功能"而不是"添加了功能"）

## 测试

- 为新功能编写测试用例
- 确保所有测试通过
- 运行测试覆盖率报告，确保覆盖率达标

## 文档

- 更新相关文档以反映您的变更
- 保持文档的准确性和完整性
- 为新功能添加使用说明

## 行为准则

- 尊重其他贡献者
- 接受建设性的反馈
- 保持开放和包容的态度
- 遵守GitHub社区规范

## 问题反馈

如果您在开发过程中遇到问题，请通过以下方式反馈：

- 创建GitHub Issue
- 描述问题的详细信息
- 提供重现步骤
- 附上相关截图或日志

感谢您的贡献！
